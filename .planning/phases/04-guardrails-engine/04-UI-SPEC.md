### 3. Add Token Bottom Sheet

Triggered by "Add Token" button. `showModalBottomSheet` with `DraggableScrollableSheet`.

### 4. Guardrail Violation Banner

## Interaction States

### Master Toggle (Guardrails ON/OFF)

| State | Visual |
|-------|--------|
| ON (default) | `Switch` value: true, primary color thumb; no subtitle |
| OFF | `Switch` value: false; subtitle "All guardrails disabled" in `BrandColors.error`; expandable whitelist card hidden |
| Toggling OFF | `showAuthChallengeDialog()` fires; if rejected, toggle reverts to ON |
| Toggling ON | No auth required; toggle switches immediately |

### Guardrail Violation Banner

| State | Visual |
|-------|--------|
| No violation | Banner not rendered (zero height) |
| Violation present | Error banner visible; Confirm button disabled |
| Override in progress | `showAuthChallengeDialog()` dialog open |
| Override accepted | Banner clears; `guardrailBypassed = true`; Confirm re-enables |
| Override rejected | Banner stays; Confirm stays disabled |

### Token Whitelist Management

| Action | Behavior |
|--------|----------|
| Add token (held) | Tap row in search results тЖТ immediate add, row appears in whitelist, sheet dismisses |
| Add token (paste) | Type/paste 44-char address тЖТ onSubmitted validation тЖТ add тЖТ sheet dismisses; invalid length shows inline error "Invalid mint address (must be 44 characters)" |
| Remove token | Tap X тЖТ immediate removal from list; undo not offered |
| Empty whitelist | "No restrictions тАФ all SPL tokens allowed" caption under card header |

## Phase-Specific Constraints

The following decisions from CONTEXT.md are locked and must not be overridden during implementation:

1. **Inline banner only** тАФ no navigation, no modal for guardrail blocks. Banner stays on review screen.
2. **One-shot bypass** тАФ `guardrailBypassed` resets on `SendState.reset()` and `SwapState.reset()`. No timed windows.
3. **Agents cannot bypass** тАФ HTTP 403 is final for agent intents. No UI bypass path exposed to agents.
4. **SOL transfers exempt** тАФ token whitelist does not apply to SOL sends. No UI distinction needed (handled in Rust engine).
5. **NFT sends exempt** тАФ no guardrail UI on NFT send screen.
6. **Program whitelist deferred** тАФ no UI for `program_whitelist` in this phase. Field exists in config, not surfaced.
7. **Empty whitelist = all allowed** тАФ settings must communicate this clearly via the "No restrictions" caption.
8. **Unknown mint display** тАФ raw truncated address only for manually pasted mints not in `balanceProvider`. No Helius DAS lookup.
