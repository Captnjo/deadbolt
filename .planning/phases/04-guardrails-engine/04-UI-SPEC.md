## Spacing Scale

Declared values (multiples of 4) ŌĆö for all new components authored in Phase 4:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps (e.g. `SizedBox(width: 4)` within Row) |
| sm | 8px | Between related inline elements, stacked form fields, tighter gaps inside new components |
| lg | 16px | Card internal padding (`EdgeInsets.all(16)`), section item spacing, gaps inside new components requiring comfortable padding |
| xl | 24px | Between major review-step sections, section title to first item |
| 2xl | 32px | Not actively used in existing screens ŌĆö reserved |
| 3xl | 48px | Not actively used in existing screens ŌĆö reserved |

New Phase 4 components must use only tokens from the table above (xs / sm / lg / xl). Use `sm` (8px) for tighter gaps and `lg` (16px) for card-internal padding.

### Codebase-inherited exceptions (not for new components)

| Value | Location | Why locked |
|-------|----------|------------|
| 12px (vertical button padding) | `elevatedButtonTheme` in `buildBrandTheme()` | Theme-level declaration; changing it affects all elevated buttons site-wide |
| 12px (section title to items gap) | `SecuritySettingsSection` (lines 44ŌĆō110) | Existing widget not modified in this phase; gap is part of its internal layout |

**Source:** Observed from `send_screen.dart` (lines 603ŌĆō698), `security_settings_section.dart` (lines 44ŌĆō110), `brand_theme.dart` (`elevatedButtonTheme`).

### 2. Token Whitelist Row

Each row in the expanded whitelist:
- Leading: `CircleAvatar` with network image (token icon) ŌĆö fallback to first letter of symbol
- Title: token symbol or "Unknown" (14px w400)
- Subtitle: truncated mint address (first 4 + "..." + last 4 chars, 13px, `BrandColors.textSecondary`)
- Trailing: `IconButton(icon: Icon(Icons.close, size: 18), tooltip: 'Remove token', onPressed: ...)` ŌĆö removes token from whitelist

**Accessibility:** The remove `IconButton` must include `tooltip: 'Remove token'` so that screen readers announce a meaningful action label. If a per-token label is preferable (e.g. "Remove BONK"), use `Semantics(label: 'Remove $symbol', child: IconButton(...))` instead.

## Interaction States

### Master Toggle (Guardrails ON/OFF)

| State | Visual |
|-------|--------|
| ON (default) | `Switch` value: true, primary color thumb; no subtitle |
| OFF | `Switch` value: false; subtitle "All guardrails disabled" in `BrandColors.error`; expandable whitelist card hidden |
| Toggling OFF | `showAuthChallengeDialog()` fires; if rejected, toggle reverts to ON |
| Toggling ON | No auth required; toggle switches immediately |

### Guardrail Violation Banner

| State | Visual |
|-------|--------|
| No violation | Banner not rendered (zero height) |
| Violation present | Error banner visible; Confirm button disabled |
| Override in progress | `showAuthChallengeDialog()` dialog open |
| Override accepted | Banner clears; `guardrailBypassed = true`; Confirm re-enables |
| Override rejected | Banner stays; Confirm stays disabled |

### Token Whitelist Management

| Action | Behavior |
|--------|----------|
| Add token (held) | Tap row in search results ŌåÆ immediate add, row appears in whitelist, sheet dismisses |
| Add token (paste) | Type/paste 44-char address ŌåÆ onSubmitted validation ŌåÆ add ŌåÆ sheet dismisses; invalid length shows inline error "Invalid mint address (must be 44 characters)" |
| Remove token | Tap X (`IconButton` with `tooltip: 'Remove token'`) ŌåÆ immediate removal from list; undo not offered |
| Empty whitelist | "No restrictions ŌĆö all SPL tokens allowed" caption under card header |

## Phase-Specific Constraints

The following decisions from CONTEXT.md are locked and must not be overridden during implementation:

1. **Inline banner only** ŌĆö no navigation, no modal for guardrail blocks. Banner stays on review screen.
2. **One-shot bypass** ŌĆö `guardrailBypassed` resets on `SendState.reset()` and `SwapState.reset()`. No timed windows.
3. **Agents cannot bypass** ŌĆö HTTP 403 is final for agent intents. No UI bypass path exposed to agents.
4. **SOL transfers exempt** ŌĆö token whitelist does not apply to SOL sends. No UI distinction needed (handled in Rust engine).
5. **NFT sends exempt** ŌĆö no guardrail UI on NFT send screen.
6. **Program whitelist deferred** ŌĆö no UI for `program_whitelist` in this phase. Field exists in config, not surfaced.
7. **Empty whitelist = all allowed** ŌĆö settings must communicate this clearly via the "No restrictions" caption.
8. **Unknown mint display** ŌĆö raw truncated address only for manually pasted mints not in `balanceProvider`. No Helius DAS lookup.
