  <action>
Add a new `pub fn sign_message(message_hex: String) -> Result<SignedTxDto, String>` function to `rust/deadbolt_bridge/src/api/sign.rs`.

Implementation:
1. Decode `message_hex` from hex to `Vec<u8>` using `hex::decode(&message_hex).map_err(|e| format!("Invalid hex: {e}"))?`
2. Get the active signer: `let mgr = manager_pub().read().map_err(|e| e.to_string())?;` then `let signer = mgr.get_active_signer().map_err(|e| e.to_string())?;`
3. Sign the raw message bytes: `let sig_bytes = signer.sign(&message_bytes).map_err(|e| e.to_string())?;`
4. Return `SignedTxDto { base64: String::new(), signature: hex::encode(sig_bytes) }` -- `base64` is empty because sign_message has no transaction, only a signature. This matches the Dart-side expectation from Plan 02 SUMMARY: "sign_message approve path returns empty base64 with signature only -- no on-chain tx submission for message signing."

The function follows the exact same pattern as `sign_serialized_transaction` -- read manager, get signer, sign, return DTO.

Note: This function goes in `sign.rs` (NOT `agent.rs`) because `sign.rs` maps to the FRB-generated `sign.dart`. Functions in `agent.rs` map to `agent.dart`. Since `agent.dart` is also FRB-generated, either would work, but `sign.rs` is semantically correct (it's a signing operation) and groups with `sign_serialized_transaction`.

Add the `hex` crate import at the top if not already present (it's already a dependency of the workspace via deadbolt_core).
